V3 Req04 / Resp03 Tool Governance Review

请求生命周期从 Client SSE Request Start 开始：server accept → request normalization → tool output pair normalization → continuation owner check → Req04 restore → current-turn merge/governance。响应侧单独是 Resp03 response governance + Resp04 continuation save。 每个节点都写清楚“干什么”和“逻辑”，HTML 由架构渲染器生成。

Canonical Markdown source: docs/architecture/wiki/v3-req04-tool-governance-review.md Request edge: Client SSE → Server raw → ReqInbound02 → Tool output pair normalization → Req03 owner check → Req04 restore/govern → ReqExecution05 Response edge: Provider raw → ProviderRespCompat02 → RespInbound02 → Resp03 text harvest/tool repair → finish_reason tool_call|stop hooks → Resp04 continuation save endpoint → RespOutbound05 → json2sse

Main Rule

  1. Request lifecycle starts at Client SSE Request Start, then server accept, request normalization, tool output pair normalization, continuation owner check, Req04 restore, current-turn merge/governance, and ReqExecution handoff.
  2. Restored context is already canonical; after restore, Req04 merges only current request deltas and current tool surfaces.
  3. This small skeleton does not declare a request-side internal artifact removal path because no confirmed requirement exists.
  4. Error feedback is preserved: parse-error, unknown-tool, unsupported, schema reject, and execution failure outputs remain model correction input.
  5. Response chain runs provider raw → compat → RespInbound normalization before Resp03. Resp03 first harvests text, completes/repairs tool frames, may correct finish_reason, then branches tool_call/stop into different servertool hooks.
  6. Provider codec owns malformed provider fields; Req04/Resp03 cannot become provider-specific workaround or error projection layer.
  7. Diagnostics stay side-channel only and must not enter provider body or client normal payload.

Request node logic

RQ00Client HTTP/SSE /v1/responses stream=true

Client SSE Request Start

干什么
客户端发出 SSE 请求；审计计时和 request lifecycle 从这里开始。
逻辑
客户端请求 intent 是 SSE，server 对这个入口后续必须按 SSE client response 语义返回；这不等同于 provider 一定 SSE。
RQ01V3Server03HttpRequestRaw

Server SSE Frame Accepted

干什么
server 接收 HTTP/SSE 请求、绑定 endpoint、request id、port/server id。
逻辑
server 只捕获原始请求和入口事实；不做工具配对、不修 history、不恢复 continuation。
RQ02V3HubReqInbound02Normalized

Request Normalization

干什么
把客户端协议请求非破坏性归一化为 Hub request normal payload。
逻辑
只做入口协议解析、stream intent、input/messages 形状归一；不得裁剪真实 payload 或把工具输出降级成文本。
RQ03function_call_output / custom_tool_call_output adjacency

Tool Output Pair Normalization

干什么
归一化客户端提交的工具输出结果，并保持它和对应 tool call 的 call_id/type 关系。
逻辑
parse-error、unknown-tool、unsupported feedback 是客户端反馈结果；这里只归一配对关系和顺序，不删除、不补造。
RQ04V3HubReqContinuation03Classified

Continuation Owner Check

干什么
校验 entry protocol、continuationOwner、session/conversation、port/group scope。
逻辑
只分类 owner/scope；不恢复 payload。owner 不匹配、entry 不匹配、scope 不匹配必须 fail-fast。
RQ05V3LocalContinuationStore::restore_at_req04

Continuation Restore at Req04

干什么
在 request Chat Process 入口恢复上一轮 Resp04 保存的 canonical local context。
逻辑
恢复的是 Resp04 已保存的 canonical context，不是重新读取一遍历史；恢复后只合并当前请求增量工具输出和工具 surface。
RQ06top-level tools + input[].additional_tools.tools

Merge Current Tool Surfaces

干什么
合并当前请求携带的 top-level tools 和 Codex additional_tools。
逻辑
保留原 surface；additional_tools 是 Codex capability declaration surface，不能为了内部工具注入而 flatten/drop。
RQ07function_call_output / custom output truth

Preserve Client Tool Feedback

干什么
把当前客户端工具执行结果并入 governed request truth。
逻辑
只根据显式协议字段配对 call_id/type；错误反馈是模型下一轮纠错输入，不能按错误文本删除。
RQ08reasoningStop / servertool request hook profile

Inject Current Internal Tools

干什么
按当前 turn policy 注入 reasoningStop 等内部工具和必要 guidance。
逻辑
最多一次；append/augment 当前 turn，不覆盖客户端工具，不清空 system/developer/user context。
RQ09V3HubReqExecution05Planned

Emit Req04 Governed Request

干什么
把 restored context + 当前请求工具 surface + 当前工具输出结果交给 ReqExecution05。
逻辑
Req04 到此结束；provider wire 字段错误只能在 ReqOutbound/provider codec 修，不能在 Req04 删除 transcript truth。

Response node logic

RS00V3ProviderRespInbound01Raw

Provider Response Raw

干什么
接收 provider 原始响应 JSON/SSE 事件。
逻辑
这里只保留 provider raw truth；不做工具治理、不解释 finish_reason、不投影 client。
RS01ProviderRespCompat02ProviderCompat

Provider Response Compat

干什么
先做 provider-specific response compat，把上游差异映射到 Hub 可解析形状。
逻辑
compat 只处理 provider 协议差异；不得承担 servertool、stopless、普通工具治理。
RS02V3HubRespInbound02Normalized

RespInbound Normalization

干什么
把 compat 后的 provider 响应归一化成 Hub response semantic。
逻辑
归一化只建立 Hub response 语义输入；finish_reason 分流和工具治理还不能在这里做。
RS03Resp03 text harvest

Text Harvest First

干什么
进入 Resp03 后先收割文本、reasoning、delta 累积片段。
逻辑
先把文本/增量收齐，避免后续工具补齐和 finish_reason 判定基于不完整响应。
RS04Resp03 tool frame completion

Complete / Repair Tool Frames

干什么
补齐/修正可从响应语义确定的 tool frame，例如 text-harvest 后形成完整 tool call。
逻辑
这个阶段可能改写 finish_reason，例如从 stop 修正为 tool_call；这是 finish_reason 分流前的最后治理准备点。
RS05finish_reason: tool_call | stop | other

Inspect finish_reason

干什么
根据补齐修复后的 finish_reason 选择响应治理路径。
逻辑
tool_call 和 stop 走不同 servertool hook；两条响应治理分支在 Resp03 内并行建模，但不和请求图混合。
RS06servertool hook under finish_reason=tool_call

Tool-call Servertool Hook

干什么
在 tool_call 分支先判断模型 tool call 是否属于 servertool/内部工具。
逻辑
servertool 拦截优先于普通工具治理；如果被 servertool 接管，就进入 servertool response governance，不再当普通 exec/apply_patch 处理。
RS07exec_command / apply_patch / client tool governance

Ordinary Tool Governance

干什么
servertool 未拦截的 tool_call 进入普通工具治理，例如 exec_command、apply_patch、客户端工具调用治理。
逻辑
这里治理普通工具 identity、arguments、call_id、required_action/client projection 输入；不得删除客户端可见工具真相。
RS08servertool hook under finish_reason=stop

Stop Servertool Hook

干什么
在 stop 分支执行 stop/servertool/stopless 相关 hook。
逻辑
stop 下的 servertool hook 和 tool_call 下的 servertool hook 是不同入口；可治理 natural stop、reasoningStop、guard/pass-through 等响应状态。
RS09StoplessCenterMetadataControl

Update Runtime Control Side-Channel

干什么
根据 tool_call/stop 分支治理结果更新 runtime control side-channel。
逻辑
只写 side-channel；不得把 stopless/servertool/debug/control 字段放进 provider body 或 client normal payload。
RS10V3HubRespChatProcess03Governed::output

Emit Resp03 Governed Semantic

干什么
合流输出 Resp03 已治理的 response semantic。
逻辑
到 Resp03 出口时响应治理完成；后续节点只能 save / project，不再解释工具语义。
RS11V3HubRespContinuation04Committed

Resp04 Continuation Save

干什么
保存 Resp03 已治理结果形成下一轮 Req04 可恢复的 canonical local context；这是 Chat Process 终点。
逻辑
Resp04 只 commit/release continuation；禁止重新解释响应、补工具、修 history 或注入 guidance。保存完成后才允许进入 RespOutbound。
RS12V3HubRespOutbound05ClientSemantic

RespOutbound Client Semantic

干什么
Chat Process 结束后，把已治理/已保存的 Hub response semantic 投影为入口协议可见的 client semantic。
逻辑
只做 client protocol projection；不保存 continuation，不恢复请求，不吞上游错误。
RS13V3ServerRespOutbound06ClientFrame / json2sse

JSON to SSE Client Frame

干什么
对客户端 SSE 入口，把 outbound client semantic 转成 SSE frame 并发送。
逻辑
这是传输 framing：JSON/Semantic → SSE frame；不得再治理工具、保存 continuation 或修响应语义。

Audit notes

split

Client SSE request lifecycle split from response governance

请求侧从 Client SSE 开始，经过 server accept、ReqInbound normalize、工具输出配对归一、continuation owner check、Req04 restore、当前请求 merge/governance；响应侧先 compat 再归一化，Resp03 内先文本收割/工具补齐，再按 finish_reason 分 tool_call/stop 两条 servertool hook 分支。两边不能混成一张图。

request

Error feedback is preserved

普通模型 tool call 和客户端 tool output 是 transcript truth。即使 output 是 parse error、unknown tool、unsupported call、schema reject 或 execution failure，也必须回给模型纠错。

request

No request-side artifact-removal path

这张小骨架不声明请求侧内部产物移除路径；没有已确认需求就不画、不实现、不锁这条边。

response

Resp03 owns response governance

响应侧 compat 和归一化之后，文本收割、工具补齐修复、finish_reason 分流、servertool hooks、普通工具治理都只能在 Resp03 做；RespOutbound/handler 不能补第二套治理。

response

Resp04 continuation save is Chat Process endpoint

Resp04 是 Chat Process 终点：只保存 Resp03 已治理的 continuation truth。之后才进入 RespOutbound，再由 json2sse 做客户端 SSE framing。

owner

Provider codec owns malformed provider fields

provider 因字段 shape 报错时，修 ReqOutbound / provider codec，目标是 provider-bound request 不再产生错误字段；不是在 Req04/Resp03 删除正常 transcript。

side-channel

Diagnostics stay side-channel only

debug、metadata、stopless runtime control、snapshot 只能走 side-channel carrier；不得混进 provider body 或 client normal payload。

Forbidden action focus

Do not repair tool continuity by deleting request truth or by moving governance to the wrong side. Request-side current tool output pair normalization belongs before/inside Req04 and must preserve client/model truth. Restored context is canonical and is not read again as raw history. Response-side tool/servertool/stopless governance belongs to Resp03, then Resp04 saves continuation truth as the Chat Process endpoint; outbound and json2sse happen after that. Provider malformed fields are fixed at ReqOutbound/provider codec; debug/control facts stay side-channel only.

# ServerTool / Stopless Lifecycle (CLI projection only)

## 1. Scope

This document defines the only valid lifecycle for:

- stop_message / stopless auto-continue
- stop schema gate / learned note write
- CLI projection for `stop_message_auto`

Design goals:

1. `stop_message_flow` does **not** reenter Hub Pipeline. It projects a client-visible `exec_command` tool call.
2. The CLI projection is the only valid execution path for `stop_message_auto`.
3. `stop_message_flow` followup hops are generated by the CLI tool's output being fed back as a normal client request; each hop is a standard request that may retrigger stopless.
4. Loop safety is enforced by `used/max_repeats` counters (Rust `stop-message-core`): provided schema = 3 consecutive stops; missing schema = 10 consecutive stops (missing does not consume budget).
5. `/goal active` and plan mode skip the stopless gate entirely.
6. Servertool has no private response protocol: CLI projection returns through the normal response chain as an `exec_command` tool result.
7. Response direction is provider/model inbound -> `HubRespInbound02Parsed` -> `HubRespChatProcess03Governed` -> `HubRespOutbound04ClientSemantic` -> client outbound.

## 2. Code Entry Points

- Stopless gateway: Rust `stop-message-core` via `native-stop-message-auto-semantics.ts`
- Budget counter: Rust `stop-message-counter.rs` (`STOP_SCHEMA_MISSING_MAX_REPEATS=10`, `STOP_SCHEMA_MAX_REPEATS=3`)
- Handler: `sharedmodule/llmswitch-core/src/servertool/handlers/stop-message-auto.ts`
- CLI projection: `sharedmodule/llmswitch-core/src/servertool/cli-projection.ts` (`buildServertoolCliProjectionForAutoFlow`)
- CLI executor: `sharedmodule/llmswitch-core/src/servertool/cli-executor.ts`
- Note write: `sharedmodule/llmswitch-core/src/servertool/memory/cache-writer.ts` (`writeStoplessLearnedNoteEntry`)

## 4. CLI Projection Contract

### 4.1 Tool Name

Only `exec_command` is projected for `stop_message_auto`. No other tool name, no server-side tool restoration, no fake tool.

### 4.2 CLI Input Fields (required)

```json
{
  "flowId": "stop_message_flow",
  "continuationPrompt": "<heuristic multi-segment audit prompt>",
  "repeatCount": N,
  "maxRepeats": M,
  "stdoutPreview": "<short summary>"
}
```

### 4.3 Continuation Prompt Must Include

The prompt must require the model to address ALL of:

1. **Goal**: What is the current user goal?
2. **Done steps**: What has been completed?
3. **Completion status**: Is the goal finished? Blocked? Still in progress?
4. **Next step**: If next step exists, execute it directly with evidence (files/logs/command output/tests).
5. **Evidence**: How to verify each claim?
6. **Root cause**: What is the likely cause of any issue?
7. **Excluded factors**: What has been ruled out?
8. **Diagnostic order**: What is the next diagnostic step?
9. **Learned**: What reusable fact / lesson was learned?

Fixed "continue execution" text is forbidden. The prompt must dynamically reference the specific situation.

### 4.4 Stop Schema (required when trying to stop)

When the model attempts to stop, it must provide this JSON at the end of the response:

```json
{
  "stopreason": 0,
  "reason": "finished/blocked reason",
  "has_evidence": 1,
  "evidence": "file/log/command output/test evidence",
  "issue_cause": "problem cause or empty string",
  "excluded_factors": "ruled-out factors or empty string",
  "diagnostic_order": "diagnostic order already executed",
  "done_steps": "completed steps or empty string",
  "next_step": "next step to execute immediately or empty string",
  "next_suggested_path": "suggested path or empty string",
  "learned": "learned fact or empty string"
}
```

- `stopreason` MUST be numeric: `0` = finished, `1` = blocked, `2` = continue_needed
- `stopreason` and `has_evidence` must be numeric; text fields only check non-empty
- Missing schema: does NOT consume budget; triggers a new followup prompt
- Provided but invalid schema (non-numeric stopreason, missing reason, etc.): consumes budget

## 5. Budget Rules

| Condition | Consumes Budget | Next Action |
|-----------|----------------|-------------|
| `finish_reason != stop` | reset | normal resp_outbound |
| missing schema | NO | new followup prompt |
| invalid schema | YES | new followup prompt |
| `stopreason=0|1` + valid reason | reset | final stop |
| `stopreason=2` + non-empty next_step | YES | new followup prompt |
| 3 consecutive provided-schema stops | exhausted | final stop summary |
| 10 consecutive missing-schema stops | exhausted | final stop summary |

Non-stop response or tool call always resets the budget regardless of consecutive count.

## 6. Memory Write Rules

- `note.md` write: ALLOWED only when `schemaGate.action=allow_stop` AND `learned` is non-empty.
- Forbidden: followup hop, invalid schema, missing schema, budget exhausted, reenter failure.
- Only write learned fact; do not write full stop summary or debug output.

## 7. What Is NOT Part of This Lifecycle

- `reenterPipeline` / `reenterFollowup` / `followup-request-builder`: **deprecated**, do not use
- `ServertoolReq04FollowupBuilt`: **deprecated**, do not reference
- `normal Hub reenter` for stopless: **deprecated**, stopless does not reenter
- `stopMessageFollowupPolicy` / `preserve_eligibility`: **deprecated**, not used in CLI projection
- tmux/client injection for `stop_message_flow`: **forbidden**

## 8. Other Servertools (vision, web_search)

- `vision_auto` and `web_search` follow their own outcome contracts (backend route reenter, not CLI projection)
- They are out of scope for this closeout plan

## 9. Observability Requirements

Required logs:

1. stopless gate evaluation:
   - decision (trigger/skip)
   - reason (triggered / stop_schema_missing / stop_schema_invalid / allow_stop)
   - used / left budget
   - armed / mode / max_repeats

2. CLI projection:
   - tool name (`exec_command`)
   - continuationPrompt length
   - repeatCount / maxRepeats

3. schema gate result:
   - action (continue / allow_stop / budget_exhausted)
   - stopreason value
   - learned field presence

4. memory write:
   - write success/fail
   - learned content length

## 10. Validation Checklist

1. Provider response reaches `HubRespChatProcess03Governed` with `finish_reason=stop`.
2. Stopless gate evaluates correctly: `/goal active` and plan mode skip.
3. Missing schema does NOT consume budget; invalid/provided schema DOES consume budget.
4. Budget resets on non-stop or tool call response.
5. `exec_command` CLI projection is returned (not chat completion / not followup request).
6. `continuationPrompt` contains heuristic multi-segment audit, not fixed "continue".
7. Stop schema parsing: only JSON with `stopreason` is a candidate.
8. `note.md` write only on `allow_stop` + non-empty `learned`.
9. Budget exhausted: final stop summary returned, no loop.
10. `/v1/responses` final response has `object=response` top-level.

# 2026-07 lessons

## L93-08 rcc release install must use normal npm install gate

- Trigger: `npm install -g artifacts/pack/rcc-<version>.tgz` fails, especially during bundled dependency lifecycle scripts, or release verification only uses offline/prefer-offline checks.
- Rule: prove the actual npm install path. The release tarball must pass `npm install -g <tgz> --prefix <tmp>` with normal npm behavior, then real global install when authorized. Check installed package is not repo-linked, `rcc-llmswitch-core` is not a symlink and has `dist`, all production dependencies are bundled, and build-only packages such as `esbuild` are absent from production dependencies and bundled tarball contents.
- Anti-pattern: using offline install as the solution, treating tarball file listing as equivalent to npm lifecycle install, or bundling every package from an accidentally polluted `dependencies` block.
- Evidence target: `scripts/verify-rcc-release-install.mjs` PASS; `npm run pack:rcc` PASS with that gate; `rcc --version` from installed package returns the packed version; installed package JSON and real paths do not contain the repo root.

## L93-07 LM Studio Responses direct must not be downgraded to chat

- Trigger: 4444 `/v1/responses` with `provider.type=responses` reports `PROVIDER_NOT_AVAILABLE` or LM Studio upstream 400 such as missing `text.format`.
- Rule: first prove config/runtime truth (`~/.rcc/provider/lmstudio/config.v2.toml`, `routecodex port status/dry-run`) and keep same-protocol Responses direct. Fix only the unique owner: VR availability/filtering in Rust if selection is wrong, or Rust `responses:lmstudio` compat if LM Studio requires a provider-wire field such as `text.format`.
- Anti-pattern: switching LM Studio to chat, adding Responses-to-Chat conversion, patching handler/SSE/direct response shells, or treating `provider.type=responses` as evidence that it should leave direct passthrough.
- Evidence target: exact old `/v1/responses` sample dry-run must select `lmstudio.key1.ornith-1.0-397b` with `providerProtocol=openai-responses` and `wouldReturnProviderNotAvailable=false`; live replay with `Accept: text/event-stream` must return HTTP 200, `response.completed`, no `event:error`, `created.model=ornith-1.0-397b`, and `created.text.format.type=text`.

## L93-06 Direct runtime metadata must be primitive control projection

- Trigger: image-bearing `/v1/responses` or multimodal direct requests fail before provider response with `JSON stringify failed: Maximum call stack size exceeded` or bare `Maximum call stack size exceeded`.
- Rule: native/direct/provider runtime bridge inputs may receive only minimal serializable control carriers. Project primitives at the direct runtime boundary: endpoint, port/scope, routing group, client request id, timeout controls, and protocol-specific direct flags.
- Anti-pattern: passing full `metadataForHub`, `metadataCenterSnapshot`, request `input`, image payload arrays, entry request bodies, or MetadataCenter-bound live objects into native JSON bridges/provider runtime metadata. Do not fix this in SSE, handler, snapshot writer, or outbound projection; SSE is transport only.
- Evidence target: unit regression with cyclic metadata plus `input_image` must prove `JSON.stringify(projected)` succeeds and forbidden live fields are absent; live `/v1/responses` image replay must reach VR/provider switching without `Maximum call stack` / `evaluateResponsesDirectRouteDecisionJson` logs.

## L93-05 Responses conversation capture must use entry payload, not provider wire body

- Trigger: `/v1/responses` logs `RESPONSES_STORE_MISSING_REQUEST_CONTEXT` after VR/provider id rewrite even though request capture/rebind code appears present.
- Rule: if server swaps `body=hubBody`, preserve the original entry body as same-request data-plane truth and let request-executor capture use it before provider wire body. Provider wire payload (`data/messages`, provider model) is not valid Responses continuation context source.
- Anti-pattern: fixing this by scope fallback, by using provider response fields, by relying on debug `contextSnapshot`, or by moving request context into MetadataCenter control state.
- Evidence target: focused capture red/green where Hub body has provider wire shape and `__raw_request_body` has Responses `input/tools`; live `/v1/responses` first turn plus `submit_tool_outputs` continuation must produce no new `RESPONSES_STORE_MISSING_REQUEST_CONTEXT` / `record.missing_request_context` logs.

## L93-01 Codex capability/tool audits need one-shot live replay

- Trigger: debugging Codex `tool_search`, MCP, browser/computer, or node_repl exposure through RouteCodex.
- Rule: first verify `/v1/models` model-level metadata, then inspect Codex source for client/deferred tool contract, then run a one-shot `codex exec -p rcm -m gpt-5.5` replay that instructs the model to call `tool_search` exactly once and stop.

## L93-02 Provider protocol mismatch is not a VR hit filter

- Trigger: priority/provider failover logs show selected backup excluded or failed because `providerProtocol` / `outboundProfile` differ, especially `openai-responses` entry selecting `anthropic-messages` or `openai-chat` targets.
- Rule: VR decides candidate hit and ordering only from route/capability/health/exclusion/default-floor truth. Direct-vs-relay protocol compatibility belongs to server direct/relay execution boundary, not VR hit acceptance or executor selection commit.
- Anti-pattern: throwing `ERR_VR_SELECTION_PROVIDER_PROTOCOL_MISMATCH` or filtering candidates because the selected provider protocol differs from the inbound/client protocol; this turns execution-mode choice into false provider unavailability.
- Evidence target: Rust VR `protocol` tests must keep cross-protocol route/forwarder targets selectable, and executor pipeline-attempt tests must accept a selected cross-protocol target while still failing fast on excluded-provider reselection.

## L93-03 Response-stage prepass/finalize payloads belong to native result plans

- Trigger: `response-stage-prepass-shell.ts` or `response-stage-finalize-shell.ts` constructs `{ action: "continue_to_execution" }`, `{ action: "return_result" }`, passthrough engine results, or returns `responseStageAutoHook.result` / `passthroughResult` directly.
- Rule: Rust `plan_servertool_response_stage_runtime_action` must return `prepassResult` and `finalizeResult`; TS may run required auto-hook IO and switch on native action discriminants only.
- Anti-pattern: moving hook matching to Rust but leaving prepass/finalize result object materialization in TS; that keeps response-stage as a second business owner.
- Evidence target: focused response-stage Jest, servertool active orchestration audit, native bridge spec, Rust servertool-core/NAPI tests, native rebuild, sharedmodule tsc, and `verify:servertool-rust-only` must all pass before committing the slice.

## L93-04 Request/usage release logs should print compact truth only

- Trigger: release logs show raw/prepared input shape JSON, `finish_reason=unknown`, `usage=in:n/a`, or a standalone `[router-direct.send] completed` line next to usage.
- Rule: request-start logs may keep scalar counts and mode, but must not print shape/debug payload summaries; usage logs should say `usage=unreported` when upstream omits metrics and omit finish_reason entirely when unavailable.
- Anti-pattern: inventing placeholder truth (`unknown`, `n/a`) or splitting one terminal request summary across `[router-direct.send] completed` plus `[usage]`, which obscures the request outcome.
- Evidence target: focused logging Jest plus live `codex exec -p rcm -m gpt-5.5` replay against 5520 and grep new server log lines for forbidden markers.

# 2026-07 lessons

## L93-17 Plain start must be the non-disruptive lifecycle probe

- Trigger: after release/build work, running `start` or `start --snap` unexpectedly sends shutdown/restart signals, or the user reports that restart/start "only stopped" a live server.
- Rule: distinguish verification probes from lifecycle replacement. Plain `routecodex start` / `rcc start` must pass `restart=false` and report existing service state without shutdown; only explicit `--restart`, `--exclusive`, or the dedicated `restart` command may replace a live process. After fixing source, check both global installed `dist` and live `/health`; installed CLI can be fixed while the running server still reports an older version until an intentional restart.
- Anti-pattern: using `options.restart !== false`, using `restart` to prove non-disruptive behavior, or letting unit tests with unmocked `fetchImpl` hit live `/shutdown`.
- Evidence target: focused CLI lifecycle Jest locks default `restart=false` and explicit `--restart=true`; `rg` in global installed `dist/cli/commands/start.js` shows `options.restart === true`; live plain `routecodex start --port <port>` reports `already_running_*` and post-command `/health` stays OK without new shutdown.

## L93-16 Deleted native wrapper source must be replaced by typed owner exports, not restored casually

- Trigger: a Rustification slice deletes or deprecates `native-chat-process-servertool-orchestration-semantics.ts`, then TypeScript shells import servertool functions from `native-servertool-core-semantics.ts` and `tsc` reports package-shim return values as `unknown`.
- Rule: keep the current Rust owner wrapper as the single TS import surface and add explicit local type signatures for package-shim functions that TS shells consume. Cast only the imported shim function to the known contract; do not cast runtime values at call sites and do not restore deleted source unless function-map/mainline explicitly requires that file as owner.
- Anti-pattern: reintroducing the deleted wrapper to satisfy imports, adding new prod TS type shim files, or spreading `as any` / `as Record<string, unknown>` through servertool shells. That creates a second bridge owner and can break rustification audits.
- Evidence target: sharedmodule `tsc` passes; `verify:servertool-rust-only` passes; focused servertool shell/Jest proves runtime still calls the native bridge; residue scan over `sharedmodule/llmswitch-core/src/servertool` shows no `.action` application branches.

## L93-15 Model-capacity provider text belongs to retryable HTTP_429

- Trigger: provider/upstream returns only text like `Selected model is at capacity. Please try a different model.` and RouteCodex needs to retry/reroute it as 429.
- Rule: normalize provider-agnostic capacity saturation text in `normalizeKnownProviderError()` to existing recoverable `HTTP_429` (`429.1000`). This is transient capacity pressure, not account quota depletion; do not map it to `INSUFFICIENT_QUOTA`.
- Anti-pattern: adding string checks in request executor/router/direct code, using provider-specific config mapping for a generic text, or treating "try a different model" as a model-not-supported/direct-return condition.
- Evidence target: catalog test proves key/status/class; policy test proves health-impacting recoverable retry with `exclude_and_reroute`; HTTP error projection test proves no-status Error objects get `code=HTTP_429` and `statusCode/status=429`. Then run the provider failure blackbox/build/live gates when the broader worktree can build.

## L93-14 rcc lifecycle must expand config port groups and trust HTTP shutdown over PID cache

- Trigger: `rcc stop` says only one configured port has no server, while `rcc start --snap` resolves a larger port group and exits with `already_running_unmanaged` on a sibling port.
- Rule: for release lifecycle commands, a matched `targetPort` inside `httpserver.ports` must expand to the full configured port group unless the caller explicitly asks for check-only/single-port dev semantics. PID cache/listener PID discovery is a hint only; `stop` must still attempt RouteCodex `/shutdown` and verify `/health` disappears when PID discovery returns empty.
- Anti-pattern: treating no PID as no server, treating guardian health/finalize network errors as stop failure after ports are already stopped, or making plain `start --snap` stop a live server by default.
- Evidence target: focused CLI tests cover stop full-group expansion, no-PID HTTP shutdown, guardian finalize non-blocking, and plain start non-disruptive behavior with explicit `--restart` opt-in. Live validation must use `/health` before/after and confirm no unexpected shutdown for plain start.

## L93-12 Release/live SSE replay must use the installed `rcc` runtime and captured provider response

- Trigger: `/v1/responses` SSE fails in live with `Responses SSE event sequence missing request_id`, while source replay passes or source already contains a suspected fix.
- Rule: replay the exact captured provider response through the installed package root that live uses (`/opt/homebrew/lib/node_modules/rcc/...` when registry command is `rcc start`). Check snake_case dispatch fields such as `request_id` / `body_text` at the Rust NAPI boundary, then install both `routecodex` and `rcc` tarballs before live restart.
- Anti-pattern: validating only repo `dist`, installing only `routecodex`, or treating `/health` on 5555 as proof that the same package root is live. 5555 may be a localPort under serverId 5520, and live may consume `rcc`.
- Evidence target: installed replay of the old `provider-response_*.json` emits `response.created` / `response.done` with the original request_id; live `/v1/responses` smoke on 5555 returns SSE events with request_id; `/health` reports the installed version for both 5520 and 5555.

## L93-13 Expired provider cooldown import must persist cleanup immediately

- Trigger: provider-health JSON still shows a tripped provider after restart even though `cooldownExpiresAt` is already in the past.
- Rule: distinguish runtime availability from stale disk state. Import may skip expired cooldowns and make the provider available, but restart recovery is incomplete unless `refresh_provider_health_from_store()` writes the pruned provider-health state back to the port session directory.
- Anti-pattern: fixing selection/VR when diagnostics already show the provider healthy, deleting provider-health by hand, or waiting for the next provider health write as cleanup.
- Evidence target: engine-level Rust test proves refresh writes `providerCooldowns: []` after pruning expired entries; live `routecodex port status <localPort> --json` shows the provider `state=healthy`, `cooldownExpiresAt=null`, and `available=true`.

## L93-11 `/v1/models` `tool_mode=code_mode_only` changes Codex tool planning

- Trigger: Codex starts printing tool-call transcripts as assistant text (`Assistant requested tool calls...`, `apply_patch [{...}]`) after a model metadata/capability change, while `apply_patch_tool_type` and tool declarations still look present.
- Rule: inspect `/v1/models` first, then compare Codex client source for how the capability is consumed. `tool_mode: "code_mode_only"` is not a harmless capability flag: Codex hides direct nested tools and exposes code-mode executors. Omit `tool_mode` unless intentionally switching that model to code-mode executor semantics.
- Anti-pattern: fixing only response text sanitizers, patching SSE transport, or treating `experimental_supported_tools` as proof that direct/native tools are still model-visible. SSE is transport; the capability source of truth is the model metadata contract plus Codex planning code.
- Evidence target: route test proves `tool_mode` absent while freeform `apply_patch`, `web_search`, parallel tool calls, and modalities remain; live installed `/v1/models` confirms the same; latest `~/.rcc/codex-samples/openai-responses/ports/<port>/` client/provider files scan clean for transcript markers; structured tool-call turns finish with `tool_calls` where expected.

## L93-10 Responses SSE timeouts must align across semantic, transport, and provider request windows

- Trigger: `/v1/responses` direct SSE fails with `UPSTREAM_STREAM_CONTENT_IDLE_TIMEOUT` on long LM Studio prefill, or a fix changes only `providerStreamNoContentTimeoutMs` / `providerStreamContentIdleTimeoutMs`.
- Rule: 15 minute semantics are valid only when all relevant windows align: `HTTP_SSE_TOTAL_MS`, provider stream headers cap, byte-idle cap, semantic no-content/content-idle timeout, and `PROVIDER_TIMEOUTS.RESPONSES` / `BASE_SERVICE_PROFILES.responses.timeout`. Verify both `routecodex` and `rcc` installed package roots, then restart/reload the live managed process and prove the new PID handles a real `/v1/responses` SSE request.
- Anti-pattern: adding per-provider hidden timeout overrides, downgrading LM Studio Responses direct to chat, or claiming closure from source constants while global installed `dist` or the live process still has old `RESPONSES=240000`.
- Evidence target: focused timeout/unit gates pass; `routecodex` and `rcc` global constants all report `900000`; live reload creates a new server PID; 4444 `/v1/responses` SSE returns `response.completed` with `model=ornith-1.0-397b`; no new matching `UPSTREAM_STREAM_CONTENT_IDLE_TIMEOUT` appears for the smoke request. Long-prefill closure still requires old failing sample replay or observing beyond the old 5 minute cutoff.

- Trigger: Codex UI shows plain assistant text like `Assistant requested tool calls: - id=call_... type=function name=exec_command arguments=...`, or client-response samples contain that text in `output_text` / `response.output_text.delta`.
- Rule: first prove whether the leak is structured tool rendering or client-visible text by checking Codex session JSONL and `~/.rcc/codex-samples/openai-responses/ports/<port>/<requestId>/client-response.json`. - Anti-pattern: declaring closure from `governResponseJson` / generic display sanitizer tests alone, patching SSE transport, or stripping every field named `arguments`. SSE is transport; tool argument payloads are structured protocol data, not display text.
- Evidence target: Rust tests for both full Responses client payload projection and SSE frame projection; native build and projection gate pass; release tarballs for both `routecodex` and `rcc` installed globally; live `/v1/responses` replay against installed port returns `response.completed` and grep for the leaked transcript/call id/command returns zero.

## L93-08 routecodex+rcc release install must use normal npm install gate

- Trigger: `npm install -g artifacts/pack/rcc-<version>.tgz` fails, release verification only uses offline/prefer-offline checks, or live managed ports still execute global `routecodex` after only `rcc` was freshly installed.
- Rule: prove the actual npm install path for both package names. The release must produce `routecodex-<version>.tgz` and `rcc-<version>.tgz`; both tarballs must pass `npm install -g <tgz> --prefix <tmp>` with normal npm behavior, then real global install when authorized. Check installed package roots are not repo-linked, `rcc-llmswitch-core` is not a symlink and has `dist`, all production dependencies are bundled, and build-only packages such as `esbuild` are absent from production dependencies and bundled tarball contents.
- Anti-pattern: using offline install as the solution, treating tarball file listing as equivalent to npm lifecycle install, bundling every package from an accidentally polluted `dependencies` block, or assuming identical semver means `routecodex` and `rcc` have the same build.
- Evidence target: `scripts/verify-rcc-release-install.mjs` PASS for both packages; `npm run pack:rcc` produces both tarballs; `routecodex --version` and `rcc --version` from installed packages return the packed version; both installed package JSON/buildInfo/real paths do not contain stale repo roots or mismatched buildTime; managed port live replay confirms the runtime consumed the synchronized `routecodex` package.

## L93-06 Direct runtime metadata must be primitive control projection

- Trigger: image-bearing `/v1/responses` or multimodal direct requests fail before provider response with `JSON stringify failed: Maximum call stack size exceeded` or bare `Maximum call stack size exceeded`.
- Rule: native/direct/provider runtime bridge inputs may receive only minimal serializable control carriers. Project primitives at the direct runtime boundary: endpoint, port/scope, routing group, client request id, timeout controls, and protocol-specific direct flags.
- Anti-pattern: passing full `metadataForHub`, `metadataCenterSnapshot`, request `input`, image payload arrays, entry request bodies, or MetadataCenter-bound live objects into native JSON bridges/provider runtime metadata. Do not fix this in SSE, handler, snapshot writer, or outbound projection; SSE is transport only.
- Evidence target: unit regression with cyclic metadata plus `input_image` must prove `JSON.stringify(projected)` succeeds and forbidden live fields are absent; live `/v1/responses` image replay must reach VR/provider switching without `Maximum call stack` / `evaluateResponsesDirectRouteDecisionJson` logs.

## L93-05 Responses conversation capture must use entry payload, not provider wire body

- Trigger: `/v1/responses` logs `RESPONSES_STORE_MISSING_REQUEST_CONTEXT` after VR/provider id rewrite even though request capture/rebind code appears present.

## L93-02 Provider protocol mismatch is not a VR hit filter

- Trigger: priority/provider failover logs show selected backup excluded or failed because `providerProtocol` / `outboundProfile` differ, especially `openai-responses` entry selecting `anthropic-messages` or `openai-chat` targets.
- Rule: VR decides candidate hit and ordering only from route/capability/health/exclusion/default-floor truth. Direct-vs-relay protocol compatibility belongs to server direct/relay execution boundary, not VR hit acceptance or executor selection commit.
- Anti-pattern: throwing `ERR_VR_SELECTION_PROVIDER_PROTOCOL_MISMATCH` or filtering candidates because the selected provider protocol differs from the inbound/client protocol; this turns execution-mode choice into false provider unavailability.
- Evidence target: Rust VR `protocol` tests must keep cross-protocol route/forwarder targets selectable, and executor pipeline-attempt tests must accept a selected cross-protocol target while still failing fast on excluded-provider reselection.

## L93-03 Response-stage prepass/finalize payloads belong to native result plans

- Trigger: `response-stage-prepass-shell.ts` or `response-stage-finalize-shell.ts` constructs `{ action: "continue_to_execution" }`, `{ action: "return_result" }`, passthrough engine results, or returns `responseStageAutoHook.result` / `passthroughResult` directly.
- Rule: Rust `plan_servertool_response_stage_runtime_action` must return `prepassResult` and `finalizeResult`; TS may run required auto-hook IO and switch on native action discriminants only.
- Anti-pattern: moving hook matching to Rust but leaving prepass/finalize result object materialization in TS; that keeps response-stage as a second business owner.
- Evidence target: focused response-stage Jest, servertool active orchestration audit, native bridge spec, Rust servertool-core/NAPI tests, native rebuild, sharedmodule tsc, and `verify:servertool-rust-only` must all pass before committing the slice.

## L93-04 Request/usage release logs should print compact truth only

- Trigger: release logs show raw/prepared input shape JSON, `finish_reason=unknown`, `usage=in:n/a`, or a standalone `[router-direct.send] completed` line next to usage.
- Rule: request-start logs may keep scalar counts and mode, but must not print shape/debug payload summaries; usage logs should say `usage=unreported` when upstream omits metrics and omit finish_reason entirely when unavailable.
- Anti-pattern: inventing placeholder truth (`unknown`, `n/a`) or splitting one terminal request summary across `[router-direct.send] completed` plus `[usage]`, which obscures the request outcome.
- Evidence target: focused logging Jest plus live `codex exec -p rcm -m gpt-5.5` replay against 5520 and grep new server log lines for forbidden markers.
